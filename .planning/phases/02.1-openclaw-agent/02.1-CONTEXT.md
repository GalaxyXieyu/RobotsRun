# Phase 02.1: Agent 统一模型与 OpenClaw 类型化重定义 - Context

**Gathered:** 2026-04-07
**Status:** In discussion

<domain>
## Phase Boundary

本 phase 不直接实现最终前端页面，而是先把后台到底“在管理什么”说清楚。

当前已经确认，现有 `OpenClawManagement.vue` 更像一个 `xiaozhi agent -> MCP address/tools/functions` 的聚合观察页；但用户真正需要的也不应该是再单独做一个“OpenClaw Agent 控制台”。更合理的产品心智是：系统里只有一种业务智能体 `Agent`，只是底层实现来源不同，分成 `native` 和 `openclaw` 两种类型。

因此本 phase 只做三件事：

1. 定义统一 `Agent` 模型与类型分流。
2. 定义哪些配置是公共字段，哪些字段只属于 `native` 或 `openclaw`。
3. 给下一阶段前端与 manager-api 落地提供稳定输入。

本 phase 不承诺：

- 重写 OpenClaw runtime 或 deploy 机制。
- 直接实现完整日志、trace、会话回放。
- 同时把 Web 和 Mobile 都做成完整控制台。

</domain>

<decisions>
## Implementation Decisions

### Locked decisions

- **D-01:** 当前已上线的 OpenClaw 页面不是最终产品方向，只是一个围绕 `xiaozhi agent` 组织的 MCP 聚合页。
- **D-02:** 系统中的顶层业务对象是统一的 `Agent`，而不是并列维护一套 server agent 和一套 OpenClaw agent 控制台。
- **D-03:** `Agent` 通过 `agent_type = native | openclaw` 区分底层实现来源。
- **D-04:** 名称、音色、头像、基础资料、状态等信息属于公共字段，应继续复用统一 Agent 配置流。
- **D-05:** `native` 类型可以编辑 prompt / function / 本地模型等配置；`openclaw` 类型不应暴露本地 prompt 编辑心智，而是显示“由 OpenClaw 侧管理”。
- **D-06:** OpenClaw 的 `runtime/account/agent` 来源不应让用户手工填写；系统应先在“设置/集成”里绑定一个或多个 OpenClaw channel，再由 Agent 表单从已绑定 channel 拉取可选项。
- **D-07:** `openclaw` 类型创建/编辑时，应先选择 channel，再通过下拉选择该 channel 下可用的 runtime/account 与 OpenClaw agent。
- **D-08:** OpenClaw agent 映射、状态同步、tools 暴露仍然需要后台关系模型，但这些是实现细节，不应该成为前台第一心智。
- **D-09:** `manager-web` 是主控制面；`manager-mobile` 暂时只保留轻量查看或后续补充，不在本 phase 内并行扩成完整控制台。
- **D-10:** Phase 3 不能再直接沿用“独立 OpenClaw 工作台”思路执行，必须改为统一 Agent 创建/编辑流中的类型化扩展。

### the agent's Discretion

- 对象名称最终是保留英文术语还是在页面上转成人话标签。
- OpenClaw 映射关系是直接挂在 `agent` 扩展表上，还是单独抽象成一张可审计 relation 记录。
- 全局 OpenClaw channel 设置页是独立放在“系统设置/集成设置”，还是放在现有插件/渠道配置域下。
- 页面是以 “新建/编辑智能体” 为主入口，还是保留独立的 OpenClaw 状态侧板作为辅助入口。

</decisions>

<gray_areas>
## Remaining Gray Areas To Discuss

1. **创建流入口**
   新建智能体时是在同一表单里直接选择 `native/openclaw`，还是先创建统一 Agent 后再二次选择实现来源。

2. **OpenClaw 类型最小必填项**
   `openclaw` 类型创建时，最少需要选择哪些字段，例如 `channel_id`、`runtime/account`、`openclaw_agent_id`、同步策略。

3. **状态展示深度**
   第一版是只展示连接状态与 tools 快照，还是还要带上最近同步时间、异常原因和回跳入口。

4. **manager-api 契约位置**
   `agent_type`、OpenClaw 映射、状态聚合是直接扩在现有 agent detail / save 接口里，还是拆成独立 OpenClaw 扩展接口；channel inventory 获取是否单独由集成设置接口提供。

5. **Mobile 边界**
   Mobile 第一版是完全只读，还是允许切换类型后的基础查看与只读状态展示。

</gray_areas>

<specifics>
## Specific Ideas

- 一个更合理的 IA 候选是：
  - `基础信息`：名称、音色、头像、状态等公共字段。
  - `类型选择`：`native` 或 `openclaw`。
  - `类型专属配置`：
    - `native`：prompt、function、模型参数。
    - `openclaw`：channel 选择、runtime/account 下拉、OpenClaw agent 下拉、状态、tools 快照。
  - `辅助状态面板`：同步状态、连接状态、最近探测结果。

- OpenClaw 相关选择流程建议固定为：
  1. 在系统设置里先绑定 OpenClaw channel。
  2. 系统拉取该 channel 下可用的 runtime/account 与 agent inventory。
  3. 在 Agent 表单里选择 channel。
  4. 再从下拉框选择 runtime/account 和 OpenClaw agent。

- 如果后台对象模型不清晰，前端很容易做成“另起一套 OpenClaw 页面”，最后让运营在两套智能体心智之间来回切换。

- 这一 phase 的输出应该让后续 Phase 3 可以明确回答：
  - Agent 创建/编辑页如何按类型动态渲染。
  - manager-api 需要如何承载 `agent_type` 与 OpenClaw 扩展配置。
  - 哪些状态先只读展示，哪些配置允许在后台直接改动。

</specifics>
