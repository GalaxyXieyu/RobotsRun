# 02.1-01 Summary

## Result

已补齐 `02.1` 的 source-of-truth 文档：

- `.planning/codebase/UNIFIED_AGENT_OPENCLAW_MODEL.md`

这份文档不再只是重述讨论结论，而是用当前 `xiaozhi-esp32-server` 已落地的实现，确认“统一 Agent + `native/openclaw` 类型化”已经成为后续二开的事实基础。

## Key Conclusions

1. 顶层业务对象仍然是统一 `Agent`，而不是并列维护一套 OpenClaw 智能体产品。
2. `openclaw` 类型的绑定流已固定为“先维护 channel，再通过 inventory 下拉选择 runtime/account/agent”。
3. `openclaw` 类型不应暴露本地 prompt 编辑；本地语义字段继续属于 `native` 类型。
4. OpenClaw channel、binding、inventory、direct chat、clear session 已经形成明确的 manager-api 契约。
5. `/openclaw-management` 应被理解为 settings/integration workspace，而不是第二套 Agent 入口。

## Implementation Evidence

- `roleConfig.vue` 已提供 `native/openclaw` 类型切换和 OpenClaw binding 面板。
- `OpenClawManagement.vue` 已具备 channel、setup-guide、inventory、direct chat、clear session。
- `OpenClawConfigController.java` 与 `OpenClawConfigServiceImpl.java` 已提供对应 API。
- `openclaw_admin_handler.py` 已在 runtime 侧聚合 inventory 并处理 direct chat / clear session。

## Why This Matters

后续二次开发如果不先接受这个模型，最容易重复犯两个错误：

- 继续把 OpenClaw 当成独立控制台对象来设计。
- 在表单里重新放出 runtime/account 的手工输入，而不是复用 inventory。

`02.1` 的价值就是把这两个错误路径提前封死。
